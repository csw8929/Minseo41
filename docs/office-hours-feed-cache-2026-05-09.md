## Problem Statement

현재 `SubscriptionRepo`는 `FeedScreen` 진입 시점에 모든 구독 채널의 RSS를 병렬 fetch함. RSS 호출이 자주 막혀 (네트워크 일시 단절 / YouTube 측 throttling / 일부 채널 503) 빈 리스트 또는 부분 누락이 발생. InnerTube를 list fetch fallback으로 쓰는 것은 신뢰성이 더 낮아 채택할 수 없음. 결과적으로 "앱을 열었는데 오늘 영상이 0개"인 깜빡임이 사용자 경험을 망침.

## What Makes This Cool

진짜 좋은 도구의 원칙: **사용자가 앱을 열 때 빈 화면이 절대 나오지 않는다.** 어제 데이터라도 표시. 네트워크 죽어도 표시. RSS가 막혀도 표시. 라이브 fetch 실패가 사용자 경험에 새지 않게 흡수하는 캐시 레이어가 핵심. 추가로 "이번 갱신에 새로 들어온 영상"을 unread 마커로 강조해 그날의 새로움도 같이 살림.

## Constraints

- 개인용 앱 (Play Store 배포 안 함)
- 단말 only — 서버 인프라 추가 없음 (pull-only 모델)
- 기존 `SubscribedChannel.maxCount` (default 15) + `windowDays` (default 1)을 prune 기준으로 재활용
- 기존 DB 인프라(`data/db/ChannelEntity`, `ChannelDao`, `WatchPositionDao`) 위에 확장
- InnerTube는 stream URL 추출 전용으로 유지. List fetch에는 절대 사용 안 함
- Android min SDK 26 / target 36 / Kotlin / Hilt / Compose

## Premises

1. **로컬 DB가 진실의 원천.** Live RSS는 DB를 갱신하는 입력일 뿐. UI는 항상 DB만 본다.
2. **Refresh 간격은 사용자 설정.** 옵션: 1h / 3h / 6h / 12h (기본 6h).
3. **InnerTube는 stream URL 전용.** 채널 list fetch 실패는 다음 refresh를 기다리고, 그 사이 사용자는 캐시를 본다.
4. **DB 스키마는 video 중심.** `videoId` PK, channel FK, dedup은 PK 충돌로 자연 처리.
5. **Prune 룰 = 채널별 `maxCount` + `windowDays`.** 새 룰을 만들지 않고 기존 모델을 재사용.
6. **Pull only.** FCM/서버 push 없음.
7. **Unread 마커.** 새 refresh로 들어온 영상은 시각적으로 구분. 사용자가 영상을 클릭(또는 피드 진입)하면 read로 전환.

## Approaches Considered

### Approach A: Minimal Viable
Room DB + WorkManager periodic worker만. Feed가 DB Flow를 구독. unread/실패 UI 없음.
- Pros: 표준 패턴, 작은 diff.
- Cons: 사용자가 명시한 디테일(unread, 간격 설정, 실패 가시성) 누락.

### Approach B: 풀 패키지 (선택)
A + (1) `isUnread` flag (2) per-channel `lastFetchAt`/`lastFetchError` (3) Settings에서 refresh 간격 선택 (4) "마지막 갱신 N분 전" + 부분 실패 인디케이터.

### Approach C: 서버 Push
Cloud Run RSS poller + FCM. 실시간성은 최고지만 개인용 앱에 인프라가 과함. RSS 차단 문제는 단말 → 서버로 위치만 이동. **기각.**

## Recommended Approach: B

전제 1·2·7 모두 코드에 직접 표현되며, A 대비 추가 비용은 작음 (Room 컬럼 몇 개 + Settings 화면 한 개 + 인디케이터 한두 개). "Boil the lake" 원칙대로 last 1%까지 같이 짠다.

### 데이터 모델

```kotlin
// 새 entity
@Entity(
    tableName = "videos",
    foreignKeys = [ForeignKey(
        entity = ChannelEntity::class,
        parentColumns = ["channelId"],
        childColumns = ["channelId"],
        onDelete = ForeignKey.CASCADE,
    )],
    indices = [Index("channelId"), Index("publishedAtMs")],
)
data class VideoEntity(
    @PrimaryKey val videoId: String,
    val channelId: String,
    val title: String,
    val thumbUrl: String,
    val publishedAtMs: Long,
    val durationSec: Int,
    val isUnread: Boolean,
    val firstSeenAtMs: Long,    // 이 단말이 처음 발견한 시각
)

// 기존 ChannelEntity 확장
data class ChannelEntity(
    @PrimaryKey val channelId: String,
    val name: String,
    val url: String,
    val windowDays: Int,
    val maxCount: Int,
    val lastFetchAtMs: Long?,        // 추가
    val lastFetchOk: Boolean?,       // 추가
    val lastFetchError: String?,     // 추가
)
```

### Refresh 흐름

```
RefreshFeedWorker (CoroutineWorker)
  ├─ 모든 ChannelEntity 조회
  ├─ 각 채널에 대해 병렬:
  │    ├─ NewPipeVideoExtractor.fetchChannelFeed(channelId)  // 기존 그대로
  │    ├─ 성공:
  │    │    ├─ INSERT OR IGNORE (dedup 자동), 새 row는 isUnread=true, firstSeenAtMs=now
  │    │    ├─ ChannelEntity.lastFetchAt=now, lastFetchOk=true, lastFetchError=null
  │    │    └─ Prune: 그 채널의 (publishedAt < now - windowDays*86400) OR (rank > maxCount) 삭제
  │    └─ 실패:
  │         └─ ChannelEntity.lastFetchOk=false, lastFetchError=e.message (캐시는 손대지 않음)
  └─ Result.success() (한 채널 실패해도 worker 자체는 성공)

WorkManager 등록:
  - PeriodicWorkRequest(intervalHours = userPref, NetworkType.CONNECTED)
  - 사용자가 간격 변경 시 enqueueUniquePeriodicWork(REPLACE)
  - 수동 refresh: OneTimeWorkRequest로 동일 worker 트리거
```

### UI 변화

- **FeedViewModel**: `videoDao.observeTodayFeed(): Flow<List<VideoWithChannel>>` 구독. 라이브 fetch 호출 제거.
- **FeedScreen**: unread 영상에 작은 dot 또는 옅은 배경색. 영상 클릭 시 그 row만 `isUnread=false`. 피드 스크롤 후 일정 시간 visible이면 read로 전환할지는 v2.
- **SettingsScreen**:
  - "갱신 간격" 라디오/세그먼트: 1시간 / 3시간 / 6시간 / 12시간
  - "마지막 갱신: 23분 전" + (실패 채널 있으면) "1개 채널 갱신 실패" 라인
  - "지금 새로고침" 버튼 (OneTimeWorkRequest enqueue)

### Migration

- 기존 SharedPreferences 채널 목록 → ChannelEntity로 일회성 복사 (앱 시작 시 if 빈 DB then seed)
- 첫 실행 시 DB 비어 있어 빈 화면이 나오는 걸 막으려면 첫 launch에 즉시 OneTimeWorkRequest enqueue

## Open Questions

1. **Unread 해제 트리거**: 영상 클릭 시점? 피드에서 N초 visible 시점? 단순화하려면 클릭 시점만.
2. **부분 실패 시 사용자에게 어디까지 노출?**: Settings 화면에만 vs Feed 화면 상단 작은 배너. 후자가 더 정직하지만 노이즈 가능.
3. **RSS 호출 간 백오프**: 한 번 refresh에서 한 채널이 실패하면 그 채널만 짧은 재시도 (1회) 할지, 아니면 다음 cycle을 기다릴지. v1은 단순화: 재시도 안 함.
4. **WorkManager 실제 호출 빈도 보장**: Doze 모드에서 1시간 간격이 실제로 1시간이 아닐 수 있음. 사용자에게 "최소 간격" 임을 표시할지.

## Success Criteria

- 비행기 모드/네트워크 끊긴 상태로 앱을 열어도 어제 fetch한 리스트가 즉시 표시됨
- 마지막 refresh 이후 새로 들어온 영상이 시각적으로 구분됨
- Settings에서 간격을 6h → 1h로 바꾸면 다음 사이클이 1시간 후에 동작
- 한 채널이 RSS 실패해도 다른 채널의 새 영상은 정상 갱신
- 화면 진입 latency < 100ms (DB Flow에서 첫 emit)

## Distribution Plan

기존 distribution 그대로 — `./gradlew installDebug` 로 단말 직접 설치. Play Store 배포 계획 없음. 별도 CI/CD 추가 없음.

## Next Steps

순서가 중요합니다. 한 단계씩.

1. **DB 스키마**: `VideoEntity`, `VideoDao`, `ChannelEntity` 컬럼 확장. Room migration 정의.
2. **RefreshFeedWorker**: CoroutineWorker로 1채널 fetch + insert + prune. 단일 채널 단위 테스트.
3. **WorkManager 등록**: SubFeedApp(@HiltAndroidApp) onCreate에서 PeriodicWorkRequest enqueue. SharedPreferences interval 변경 시 reschedule.
4. **Migration seed**: SharedPreferences 채널 목록 → ChannelEntity 일회성 복사.
5. **FeedViewModel 전환**: 라이브 fetch 제거, DB Flow 구독으로 교체. 첫 실행 시 OneTimeWork enqueue.
6. **Unread 마커 UI**: VideoEntity.isUnread를 FeedScreen에서 dot/배경으로 표시. 클릭 시 read 처리.
7. **Settings 화면**: 간격 선택 + "마지막 갱신 N분 전" + 부분 실패 인디케이터 + "지금 새로고침" 버튼.
8. **검증**: 비행기 모드 + 강제 refresh 실패 케이스 + 간격 변경 시 다음 사이클 시점 확인.

각 단계가 독립적으로 commit할 수 있는 단위입니다. 한 단계씩 PR 분리해도 좋고, 7번 직전까지 묶어도 무방.

### 빌드 중 발생한 에러 및 수정

**에러 1**: `hiltJavaCompileDebug FAILED — Unable to read Kotlin metadata due to unsupported metadata version`
- 원인: Hilt 2.52 가 Kotlin 2.1.0 + `@HiltWorker` (androidx-hilt-compiler 1.2.0)가 생성한 메타데이터를 읽지 못함
- 수정: `gradle/libs.versions.toml` — `hilt = "2.52"` → `"2.56.2"`

**에러 2**: `kspDebugKotlin FAILED — DaggerSuperficialValidation NullPointerException on MIGRATION_1_2 field`
- 원인: KSP 가 `@Module` `object` 내의 anonymous `Migration` 타입 필드를 resolve하려다 NPE
- 수정: `MIGRATION_1_2`, `MIGRATION_2_3` 를 `DatabaseModule` object 내부에서 파일 최상위 `private val` 로 이동

## What I noticed about how you think

- "innertube는 사실 별로인 fallback이야" — fallback 패턴에 대한 회의가 명확합니다. 안전을 늘린다고 신뢰성이 늘지 않고 오히려 복잡도만 더하는 경우를 본능적으로 거르시는 게 보입니다. 단순한 "한 번 더 시도" 가 안전 망이 아니라 새 실패 표면이라는 인식.
- "채널 리스트에 맥스와 기간이 지정되어 잇어. 그것이 기준이 되면 될 듯" — 이미 있는 모델 자산을 새 기능의 기준으로 묶었습니다. 새 룰을 만들고 싶어하지 않는 systems sense. 시스템 일관성에 대한 감각이 강해서, 기능을 더할 때 "어디에 매달까"부터 보십니다.
- "사용자가 refresh로 받아온건지 기존에 있던 건지 구분되면 좋을 듯" — UX 디테일에 민감하십니다. 데이터 캐싱 같은 인프라 결정에서 "그래서 화면에서 어떻게 보이지?" 가 자연스럽게 따라오는 분이 의외로 적습니다. 본인이 사용자라서 나오는 통찰.
- "사용자가 interval 설정할 수 있도록 하자. 간격(1/3/6/12) 정도" — 정해진 답으로 닫지 않고 사용자에게 위임하는 방향. 본인이 매일 쓰는 도구이기 때문에 "내가 그날 컨디션 따라 다르게 쓰고 싶다"가 자연스럽게 나오는 듯합니다.
