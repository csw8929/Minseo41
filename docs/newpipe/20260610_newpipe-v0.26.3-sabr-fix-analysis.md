# NewPipe v0.26.3 — SABR 우회(visionOS 클라이언트)로 해결된 문제 분석

작성일: 2026-06-10
관련 브랜치: `fix/newpipe-v0.26.3-2026-06`
업데이트: NewPipe Extractor v0.26.2 → v0.26.3 (릴리스 2026-06-09)
검증: 폴드/플립 설치 후 사용자 확인 — "업데이트 이전까지 문제가 굉장히 많았는데 대부분 해결됨"

## 결론 한 줄

그동안 겪던 문제(360p 고정, chunk 403, 라이브 파싱 실패)의 공통 뿌리는 **YouTube의 SABR 강제 적용**이었고, v0.26.3은 아직 SABR이 강제되지 않은 **visionOS 클라이언트**(Apple Vision Pro)를 새 스트림 소스로 추가해 이를 우회했다. visionOS 스트림 URL은 **PoToken 없이도 chunk fetch가 동작**하므로, 우리가 진단했던 "PoToken context 영구 불일치" 문제까지 함께 비켜간다.

## SABR이란

SABR(Server-side Adaptive BitRate)은 YouTube가 기존 URL 기반 `adaptiveFormats`를 대체하기 위해 미는 새 스트리밍 프로토콜. 클라이언트가 개별 스트림 URL을 받는 대신 `serverAbrStreamingUrl` 하나로 서버가 화질을 제어한다. SABR이 강제된 클라이언트는 기존 방식의 adaptive 스트림을 못 받고 **360p muxed만** 받게 된다.

- 업스트림 이슈: TeamNewPipe/NewPipe#13320 "[YouTube] Only MP4 360p/no separate audio tracks available (SABR enforcement)" (2026-03-08 등록)
- 우리 진단 로그에도 흔적이 있었다 — `docs/20260516_innertube-adaptive-formats-fix.md`의 streamingData 진단:
  `keys=[expiresInSeconds, adaptiveFormats, aspectRatio, serverAbrStreamingUrl]` ← 이미 5월에 SABR 키가 응답에 등장.

## v0.26.3 핵심 변경 (코드 레벨)

`v0.26.2...v0.26.3` diff 확인 결과:

1. **`ClientsConstants.java`** — VISIONOS 클라이언트 상수 추가:
   `clientId=101, clientName=VISIONOS, version=1.02, deviceModel=RealityDevice14,1`
2. **`YoutubeStreamHelper.getVisionOsPlayerResponse()`** — visitorData를 InnerTube에서 받아와 player 요청. PoToken 파라미터 자체가 없음.
3. **`YoutubeStreamExtractor`**:
   - `onFetchPage`에서 ANDROID/IOS에 더해 **visionOS를 항상 fetch** (실패해도 무시, 비필수).
   - `getHlsUrl()` 우선순위: **visionOS → IOS → ANDROID** (라이브에서 audio/video 분리 HLS 제공).
   - `getItags()` 스트림 소스: ANDROID → **visionOS(poToken=null)** → IOS.
4. **`StreamInfo.extractStreams`** — audio/video 스트림 0개여도 `dashMpdUrl`/`hlsUrl` 있으면 통과 (라이브 구제).

## SubFeed 영향

- 스트림 경로 우선순위(DASH → HLS → DASH-INLINE → MUXED)는 그대로. visionOS adaptive 스트림이 들어오면서 `DASH-INLINE` 1080p 경로와 `type=HLS` 경로가 부활.
- `SubFeedPoTokenProvider`/`PoTokenWebView`는 변경 불필요 — visionOS 경로는 PoToken을 아예 안 쓴다.
- API breaking change 없음 — `libs.versions.toml` 한 줄 수정만으로 빌드 성공.
- `20260529_youtube-potoken-quality-wall.md`의 결론("이 아키텍처로는 인앱 고화질 불가")은 **v0.26.3 시점에서 일부 무효화** — yt-dlp 방식 자체 구현 없이 NewPipe 업데이트만으로 고화질 경로가 복구됐다.

## 한계 / 위험 신호 (다음 깨짐 예측)

- **visionOS도 임시 우회다.** ANDROID_VR이 그랬듯, YouTube가 SABR 강제를 visionOS로 확대하면 같은 증상(360p 고정)이 재발한다. 그때는 다시 `/sync-newpipe`.
- **made-for-kids 영상 미지원** — visionOS 클라이언트는 키즈 영상에서 audio/video-only 스트림을 못 받음 (업스트림 PR 명시). 해당 영상은 ANDROID/IOS 경로로 폴백되므로 기존 수준(360p 또는 실패).
- **unlisted 영상** — 업스트림 테스트(`YoutubeStreamExtractorUnlistedTest`)가 실패 상태로 머지됨 (업스트림도 원인 미조사). 일부 비공개 링크 영상에서 이상 동작 가능.
- PoToken 미요구 무토큰 경로라는 점에서 YouTube 입장에선 "구멍"이므로 수명이 길지 않을 수 있음.
