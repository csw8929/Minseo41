## Problem Statement

매일 구독 채널의 새 영상들을 광고 없이 한 화면에서 재생하고, 여러 Android 기기
(폰 2대 + 태블릿) 사이에서 시청 위치를 이어볼 수 있는 개인용 Android 앱.

## What Makes This Cool

- 유튜브 앱을 열지 않고도 오늘 올라온 구독 영상만 필터링해서 볼 수 있다
- 광고 없이 재생된다 (NewPipe Extractor 기반 직접 스트리밍)
- 폰에서 보다가 태블릿에서 그 자리부터 이어본다

## Constraints

- Android 전용 (PC 불필요)
- 개인 사용 (배포/스토어 출시 계획 없음)
- YouTube ToS 위반 가능성 인지하고 진행
- YouTube 구독 목록은 YouTube Takeout export → NewPipe import 방식으로 초기 설정

## Premises

1. NewPipe Extractor 라이브러리가 YouTube 콘텐츠 추출을 담당한다.
   (직접 YouTube API 대신 사용 → 광고 없음, 할당량 없음)
2. 기기 간 이어보기는 Firebase Firestore(무료 플랜)로 동기화한다.
3. 구독 채널 목록은 Google 계정 연결 없이 YouTube Takeout 데이터를 앱에 import해서 쓴다.
4. 다운로드 기능은 NewPipe Extractor의 stream URL을 DownloadManager로 저장한다.
5. 광고 제거는 NewPipe Extractor가 직접 스트림 URL을 추출해서 ExoPlayer로 재생하는 방식으로 자연히 해결된다.

## Approaches Considered

### Approach A: NewPipe Extractor + Firebase Sync (선택)
Kotlin + Jetpack Compose로 Android 앱 작성.
NewPipe Extractor로 구독 채널 최신 영상 목록 및 스트림 URL 추출.
ExoPlayer로 광고 없이 재생. Firebase Firestore(Google Sign-In)로 {videoId, positionMs} 동기화.
Effort: L (5-10일), Risk: Med-High (NewPipe Extractor YouTube 변경에 취약)

### Approach B: 처음부터 직접 구현
Python 백엔드 (yt-dlp + FastAPI) + Android Kotlin 클라이언트 앱.
Effort: XL (2-4주), Risk: High (서버 인프라 필요)

## Recommended Approach

**Approach A** — NewPipe Extractor가 핵심 난제(광고 없는 스트리밍, 다운로드)를
이미 해결해 주므로 크기가 작고 안정적. Firebase Free Tier는 개인 사용에 충분.

### 기술 스택

```
Android:
  - Language:     Kotlin
  - UI:           Jetpack Compose
  - Video player: ExoPlayer (Media3)
  - YT extract:   NewPipe Extractor (Java/Kotlin library)
  - Sync:         Firebase Firestore (Google Sign-In 사용 — 기기 교체 시 데이터 보존)

빌드:
  - Android Studio (최신)
  - Gradle
```

### 핵심 구현 순서

1. NewPipe Extractor Gradle dependency 추가
   ```groovy
   // settings.gradle
   maven { url 'https://jitpack.io' }
   // build.gradle (app)
   implementation 'com.github.TeamNewPipe:NewPipeExtractor:v0.24.2'
   ```
2. 구독 채널 import UI (YouTube Takeout XML 파일 읽기 + 수동 채널 URL 추가 지원)
3. 각 채널의 최신 영상 목록 병렬 조회 (Coroutine IO dispatcher에서 `async {}` + `awaitAll()`)
   — ChannelExtractor는 main thread 금지. 50채널 기준 30초 이내 목표.
4. 오늘 날짜 필터링 — 기기 로컬 타임존 기준 publishedDate 비교
5. 영상 탭 → StreamExtractor → ExoPlayer 재생
6. Firebase Firestore에 positionMs 저장 (30초 debounce로 쓰기 절약)
   — WatchPosition 모델: {videoId: String, positionMs: Long, updatedAt: Timestamp}
   — 충돌 정책: 더 큰 positionMs 값 우선 (마지막으로 더 많이 본 위치)
7. 앱 시작 시 마지막 위치에서 resume
8. 다운로드: DownloadManager + stream URL → 로컬 파일 URI로 ExoPlayer 재생 (오프라인 경로 분기)

### 핵심 모듈 구조

NewPipe Extractor는 인터페이스 뒤에 추상화 — 버전 업그레이드 시 `NewPipeVideoExtractor.kt` 하나만 교체.

```
app/
  ui/
    FeedScreen.kt              — 오늘의 구독 영상 리스트
    PlayerScreen.kt            — ExoPlayer 재생 화면
    SettingsScreen.kt          — 구독 채널 import
  data/
    VideoExtractor.kt          — interface: getChannelFeed(), getStreamUrl(), download()
    NewPipeVideoExtractor.kt   — VideoExtractor 구현체 (NewPipe Extractor 사용)
    SubscriptionRepo.kt        — VideoExtractor 주입받아 사용 (DI via Hilt)
    SyncRepo.kt                — Firebase Firestore 읽기/쓰기
  model/
    VideoItem.kt
    WatchPosition.kt           — {videoId: String, positionMs: Long, updatedAt: Timestamp}
```

NewPipe Extractor가 업데이트되거나 대체 라이브러리로 바꿀 때:
- `VideoExtractor` 인터페이스는 그대로
- `NewPipeVideoExtractor.kt`만 수정하거나 새 구현체로 교체
- 나머지 코드 변경 없음

## Open Questions

1. **NewPipe Extractor 불안정성** — YouTube 업데이트 후 수일~수주 피드 조회 불가 사례 있음.
   이 앱의 핵심 의존성이므로 fallback 없음 (명시적 제약). GitHub Releases 구독 권장.
   Risk: Med-High (YouTube 정책 변경에 취약).
2. **다운로드 폴더 권한** — Android 10+ 미디어 스토리지 권한(READ/WRITE_EXTERNAL_STORAGE 또는
   MediaStore API) 처리 필요. Android 13+ 는 READ_MEDIA_VIDEO 권한 필요.

## Success Criteria

- [ ] 구독 채널 import 후 오늘 영상 목록이 30초 이내에 로드됨 (최대 50채널 기준)
- [ ] 영상 재생 중 광고 없이 재생됨
- [ ] 폰A에서 3분 시청 후 폰B에서 열면 3분부터 시작됨
- [ ] 영상 다운로드 후 오프라인 재생 가능

## Distribution Plan

개인 사용 — Play Store 출시 없음.
**GitHub Releases에 서명된 APK 업로드 후 세 기기에 직접 설치** (권장).
USB 디버깅은 개발 중에만 사용. 업데이트 시 APK 재설치.

## Dependencies

- Jetpack Media3 / ExoPlayer

## What I noticed about how you think

- PC 없이 Android만 쓴다고 했을 때 망설임 없이 방향을 바꿨다.
  "다수 기기를 고려하면 웹이 낫다"는 추천을 "PC 계획 없는데?"라고 정확히 짚어낸 것.
  기능 요구사항보다 실제 사용 패턴을 더 중요하게 본다.
- 처음 아이디어를 낼 때 "가능할지 모르겠지만"이라고 했지만,
  ToS 위반 전제를 들었을 때 "동의, 계속"이라고 바로 결정했다.
  리스크를 이해하면 주저하지 않는 스타일.
- NewPipe 제안을 들었을 때 "처음부터 만들기" 대신 "이어보기만 추가"를 골랐다.
  이미 있는 것을 활용하는 실용적인 판단.
