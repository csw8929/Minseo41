# 재생 동작 Before / After (v1.3 → v1.4)

작성일: 2026-06-14
대상 독자: 동작 흐름을 한눈에 이해하고 싶을 때. 코드 몰라도 읽힘.

---

## 0. 한 줄 요약

- **안 쓰던 부품(WEB PoToken)을 떼어냈고**, **인앱 재생에 "NAS로 뽑아오기" 옵션이 새로 생겼다.**
- 평소 동작(기본값)은 **그대로**. 새 옵션은 설정에서 켤 때만 작동한다.

---

## 1. 재생 방식은 원래도, 지금도 "설정에서 고르는 것"

```
설정 → 재생 방식
│
├─ [YouTube 앱]   ← 기본값. 영상 누르면 그냥 YouTube 앱이 열림 (광고 O, 제일 안정)
│
└─ [인앱 플레이어] ← 앱 안에서 광고 없이 재생
       │
       ├─ NAS 추출 OFF  ← 기존 방식 (NewPipe로 직접 추출)
       │
       └─ NAS 추출 ON   ← ★이번에 새로 생긴 것 (NAS가 대신 뽑아줌)
```

> "NAS 추출"은 [인앱 플레이어]를 골랐을 때만 보이는 **개발자 하위 옵션**이다.
> 사용자가 보는 큰 선택지(YouTube 앱 / 인앱)는 안 바뀌었다.

---

## 2. "인앱 재생"이 안에서 어떻게 도는가 — Before vs After

### BEFORE (v1.3) — 인앱 재생

```
 폰 (SubFeed 앱)
  │
  │ 1) NewPipe Extractor 가 영상 정보 추출
  │      - WEB PoToken 을 WebView+BotGuard 로 생성  ← 만들기는 하는데
  │                                                    실제론 아무 데도 안 쓰임 (헛수고)
  │      - visionOS 클라이언트로 스트림 주소 받음
  │
  │ 2) ExoPlayer 가 그 주소(googlevideo)에서
  ▼      영상 조각을 직접 내려받아 재생
 [ YouTube googlevideo 서버 ] ───▶ 폰에서 바로 재생
```

문제점: WEB PoToken 생성기가 **첫 재생 때 WebView를 띄워 워밍업**하느라 무겁기만 하고,
정작 그 토큰은 안 쓰였다(종류가 안 맞아서). = 죽은 코드.

### AFTER (v1.4) — 인앱 재생, "NAS 추출 ON"일 때

```
 폰 (SubFeed 앱)                         집 NAS (시놀로지, Docker)
  │                                       │
  │ 1) "이 영상 좀 뽑아줘"                  │
  │    GET /extract?v=영상ID  ───────────▶ │  yt-dlp 가 추출 (android_vr)
  │                                        │  → 재생용 주소 1개 만들어 돌려줌
  │ ◀───────────────────────────────────  │  { streamUrl: NAS주소/media/토큰 }
  │
  │ 2) ExoPlayer 가 그 "NAS 주소"로 재생 요청
  │    GET /media/토큰  ─────────────────▶ │  NAS 가 googlevideo 에서 받아
  │                                        │  ───▶ [googlevideo] 
  │ ◀──── 영상 바이트 중계 ──────────────── │  ◀───  그 바이트를 폰으로 그대로 흘려줌
  ▼
 폰에서 재생
```

핵심 차이: 추출도 **NAS가 하고**, 영상 데이터도 **NAS를 거쳐서** 온다.
(WEB PoToken 부품은 아예 없어졌다.)

---

## 3. 왜 NAS가 영상을 "직접 주지 않고 거쳐서" 주나?

YouTube는 추출할 때 발급한 통행증(PO토큰)을 **"뽑은 그 기기"에 묶어둔다.**
그래서 NAS가 뽑은 주소를 폰이 직접 들고 가면 "넌 누구냐"며 막힌다.

```
✗ 직접 주기:
   폰 ───(NAS가 뽑은 주소)───▶ googlevideo   →  거부 (다른 기기/IP)

✓ NAS가 중계:
   폰 ──▶ NAS ──▶ googlevideo                →  통과 (뽑은 NAS가 그대로 받아 전달)
        ◀──      ◀──
```

대신 대가가 있다 → **집 인터넷 업로드 속도가 곧 화질·동시시청 한계.**
(영상이 googlevideo → 우리집 NAS → 폰 으로 두 번 흐르니까.)

## 4. 무엇이 우선이고, 실패하면 어디로 가나 (폴백)

```
영상을 탭한다
│
├─ 재생 방식 = YouTube 앱 ?  ── 예 ─▶ YouTube 앱 열기 (광고O, 끝)
│
└─ 인앱 플레이어
     │
     ├─ NAS 추출 ON ?
     │     │
     │     ├─ NAS 추출 성공 ─▶ NAS 경유 재생 (360p)
     │     │
     │     └─ NAS 실패(꺼짐/오류/타임아웃)
     │              │
     │              ▼
     └────────▶ NewPipe 추출 시도
                   │
                   ├─ 성공 ─▶ 인앱 재생 (visionOS)
                   │
                   └─ 실패 ─▶ "브라우저로 열기" 등 기존 폴백
```

즉 **NAS가 죽어도 앱은 안 죽는다.** 자동으로 NewPipe로, 그것도 안 되면 외부로 넘어간다.

---

## 5. 떼어낸 것 — WEB PoToken (정리)

```
삭제한 부품:
   data/potoken/ (WebView, BotGuard 호출, po_token.html …)
   + SubFeedPoTokenProvider

왜:
   우리가 만들 수 있는 토큰(WEB용)과
   NewPipe가 실제 쓰는 토큰(visionOS/Android용)이 종류가 달라서
   → 만들어도 한 번도 안 쓰였다.

효과:
   - 코드 단순해짐 (빌드/유지보수 부담 ↓)
   - 첫 재생 때 WebView 워밍업 사라져서 약간 가벼워짐
   - 동작은 그대로 (어차피 안 쓰던 거라 잃은 기능 없음)
```

## 6. 화질 현실 (지금 v1)

| 경로 | 화질 | 비고 |
|---|---|---|
| YouTube 앱 | 풀화질 | 광고 있음 |
| 인앱 NewPipe (visionOS) | 고화질 가능 | YouTube 정책 바뀌면 주기적으로 깨짐 |
| 인앱 NAS (android_vr) | **360p** | 지금은 360p 고정. 안정적이지만 화질 낮음 |

> NAS 경로가 지금 360p인 이유: 영상+소리가 **합쳐진 단일 파일**(progressive)만 쓰는데,
> 그걸 주는 클라이언트(android_vr)가 360p까지만 주기 때문.
> 고화질은 **영상/소리 분리본을 NAS에서 합치는 v2 작업**이 필요하다.

---

## 7. NAS 경로의 진짜 가치 (솔직히)

- 지금 NAS(android_vr)는 NewPipe가 쓰는 visionOS와 **같은 가문**이라,
  YouTube가 막으면 **둘이 같이 깨진다** → "내구성 보험"으로서의 효과는 제한적.
- 진짜 보험·고화질은 **v2 (mweb + 분리스트림 합치기)**. 그건 추후 작업.
- 그래도 v1으로 **전체 배관(추출→중계→재생)이 실제로 동작함을 확인**한 게 이번 성과.
