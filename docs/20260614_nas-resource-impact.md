# NAS 추출 프록시 — 성능/전력 영향 정리

작성일: 2026-06-14
관련: `20260614_playback-flow-before-after.md`, `20260613_nas-ytdlp-extractor.md`

## 한 줄 결론

평소엔 거의 영향 없고, 재생 중에만 가볍게 쓰며, **디스크(HDD)에는 영향이 없다.** 걱정 안 해도 되는 수준.

---

## 1. 평소 (아무도 안 볼 때) — 거의 0

NAS는 어차피 24시간 켜져 있고, 떠 있는 컨테이너는 대기 중엔 **CPU ~0%**.

| 컨테이너 | 대기 메모리 | 비고 |
|---|---|---|
| `subfeed-extractor` | 약 50~150MB | FastAPI/uvicorn 프로세스가 가만히 대기 |
| `bgutil-provider` | 약 50~100MB | **현재 android_vr 모드라 미사용** → 꺼도 됨(4번) |

- 전력: 이미 켜진 NAS에 가벼운 프로세스 몇 개 = 측정 거의 안 되는 수준(보통 1~3W 미만).
- `restart: unless-stopped` 라 NAS 부팅 시 자동 기동, 항상 상주.

---

## 2. 재생 중 (영상 볼 때) — 가볍게 사용

- **추출(`/extract`)**: yt-dlp 가 영상당 **몇 초** CPU 한 코어를 잠깐 사용. 끝나면 0.
- **중계(`/media`)**: googlevideo 에서 받아 폰으로 흘려보냄 → **주로 네트워크, CPU 가벼움**. 영상 보는 내내 네트워크 활성.

진짜 "비용"은 NAS 하드웨어가 아니라 **집 인터넷 업로드 대역폭** (영상이 YouTube→NAS→폰 으로 두 번 흐름).

---

## 3. 디스크(HDD)에는 영향 없음 ← 중요

설계상 영상을 **메모리로 통과(stream)만** 하고 **디스크에 저장하지 않음.**

- HDD 쓰기 없음 = **수명 영향 없음**
- 잠자던 디스크를 깨우지 않음 (NAS 전력/수명에서 가장 민감한 HDD 스핀업이 없음)

---

## 4. 지금 당장 아낄 수 있는 것 — 놀고 있는 bgutil 끄기

현재 `android_vr` 모드라 `bgutil-provider` 는 **미사용인데 메모리만 점유**. SSH 에서 정리 가능:

```bash
sudo docker stop bgutil-provider
sudo docker rm bgutil-provider
```

> v2(mweb) 로 갈 때 `docker-compose.yml` 의 `bgutil-provider` 주석을 풀고 다시 띄우면 된다.

---

## 5. 실제로 재보는 법 (SSH)

추측 대신 숫자로:

```bash
sudo docker stats --no-stream
```

→ 컨테이너별 **CPU% / 메모리**가 표로 출력. 대기 중 1번, 영상 재생 중 1번 찍어 비교하면 정확.

---

## 요약 표

| 상황 | CPU | 메모리 | 디스크 | 네트워크 | 전력 |
|---|---|---|---|---|---|
| 대기 | ~0% | 100~250MB 상주 | 없음 | 없음 | 무시 가능 |
| 추출 순간 | 한 코어 몇 초 | +α | 없음 | YouTube 조회 | 잠깐 |
| 재생 중 | 가벼움 | 낮게 유지 | **없음** | 업로드 대역폭 점유 | 보통 |
